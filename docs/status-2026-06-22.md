# DIDSA-CAD Status Summary — 2026-06-22

## What this covers

Stage 7f: "Reference planes, triad, and plane selection" — closing three gaps found
during real-device testing of Stage 7 (3D viewport): reference planes weren't
visible, there was no XYZ orientation triad in either the 3D viewport or the 2D
sketch canvas, and there was no way to see or choose which plane a Sketch is on.
No Extrude feature, no custom/arbitrary planes, and no constraint-solver changes
were in scope.

## What's been done

| Area | Summary |
|---|---|
| Reference planes (`reference_planes.dart`) | XY/XZ/YZ rendered as 20×20-unit translucent rectangles in the 3D viewport, visible by default whenever the Part has no geometry yet. Standard colour convention (X=red, Y=green, Z=blue) carried through to which axis pair tints each plane. |
| Ray-plane hit-testing (`reference_planes.dart`) | Analytic per-axis intersection of a `Camera.screenPointToRay` ray against the three fixed planes, respecting the finite rendered rectangle (not the infinite plane), with closest-to-camera-origin winning on overlap. |
| XYZ triad, 3D viewport (`triad.dart`) | Screen-space overlay, bottom-left corner, drawn every frame after `scene.render` so it's always on top and legible from any orbit angle — chosen over a world-space object specifically for that "always visible/legible" guarantee. Each axis labelled. |
| XYZ/plane indicator, 2D sketch canvas (`plane_indicator.dart`) | Small label (`XY`/`XZ`/`YZ`) plus a 2-axis arrow pair, bottom-left of `SketchCanvas`, in the same colour convention. Renders nothing until `SketchController.plane` resolves. |
| Plane-tap interaction (`part_viewport.dart`, `part_screen.dart`, `part_toolbar.dart`) | Tapping a rendered plane rectangle highlights it (brighter tint) and slides in the toolbar with a "New Sketch on `<plane>`" action; confirming it creates the `SketchFeature` on that plane and navigates to the 2D canvas. Tapping empty space dismisses the toolbar and clears the selection. The original `+` FAB is unchanged and still defaults to XY. |
| `SketchController.plane` | New getter surfacing `SketchDto.plane` (the DTO already had it), populated in `_adoptSketchDto`, feeding the new 2D indicator. |
| Backend | No changes needed — `Plane` enum/API already accepted exactly `"XY"`/`"XZ"`/`"YZ"`, and `DocumentApiClient.createSketchFeature` already took an optional `plane` param. |

## Comments, limitations, and explicit design notes

The task asked for these to be called out explicitly rather than silently routed around:

- **Transparency**: done via `flutter_scene`'s built-in `UnlitMaterial.alphaMode = AlphaMode.blend` + `baseColorFactor` alpha — no custom shader or material. **Not yet visually verified**: how the three overlapping translucent rectangles depth-sort against each other and against real Part geometry has only been exercised through pure-math hit-testing, not a live render (see Testing limitations below). Worth a real-device look before relying on the visual result.
- **Screen-tap → ray**: built directly from `Camera.screenPointToRay(Offset, Size)`, the same call `flutter_scene` itself uses for picking — deliberately reused rather than hand-rolling screen-to-world unprojection.
- **Triad placement (screen-space vs. world-space)**: chosen as a screen-space overlay per the task's own stated preference, because a world-space triad object would foreshorten, occlude, or rotate out of clear view depending on orbit angle, defeating the "visible from all camera orientations" requirement. Its axis-to-screen-direction projection was hand-verified against `flutter_scene`'s actual internal view-matrix convention (`right = up.cross(forward)`, not the more conventional `forward.cross(up)`) — confirmed by deriving the column-major matrix by hand, not just by trusting empirical output — so the triad always agrees with what the camera will actually render, including which screen direction +X projects to.
- **2D indicator**: deliberately not a full 3D triad — just a fixed (non camera-derived) 2-axis label, since `SketchCanvas` is a flat 2D plane with no camera to derive directions from.

## Testing

**Unit tests** (pure math, fully reliable in this headless sandbox):
- `reference_planes_test.dart` — 10 tests: `ReferencePlaneKindX.localTransform` rotation derivation (×3), and `hitTestReferencePlanes` (straight hit, miss-by-direction, miss-outside-rectangle, hit-just-inside, miss-behind-origin, closest-of-multiple-planes tie-break, custom `halfSize`).
- `triad_test.dart` — 3 tests: axis label/colour order, the -Z-looking/+Y-up screen-projection convention (the hand-verified sign check above), and camera-translation independence.
- `sketch_controller_test.dart` — added a `plane` assertion alongside the existing origin-point coverage.
- `plane_indicator_test.dart` — new, 4 tests: renders nothing for a null plane, renders the correct label for each of XY/XZ/YZ.

**Widget/integration tests** (`part_screen_test.dart`, 2 new tests):
- Tapping a reference plane opens the toolbar with a "New Sketch on `<plane>`" action; confirming it creates the `SketchFeature` with the right plane and navigates to the Sketch screen.
- Tapping the viewport background dismisses the toolbar and clears the plane selection.

These two tests drive `PartViewport`'s `onPlaneTap`/`onBackgroundTap` callbacks directly (`tester.widget<PartViewport>(...).onPlaneTap(...)`) rather than through a real screen gesture — standing in for "the 3D viewport reported a tap on plane X", which the task explicitly allowed ("mocked camera/viewport acceptable").

**Testing limitation, disclosed rather than worked around silently**: a dedicated test that drove a real gesture through a live `PartViewport`/`Scene` (to exercise the actual ray-cast end-to-end) was attempted and then abandoned. It intermittently threw `Exception: Flutter GPU requires the Impeller rendering backend, but Impeller is not enabled` — asynchronously, from inside `Scene` construction, sometimes attributable to a different test's run window than the one that triggered it. This is a pre-existing limitation of this headless sandbox (consistent with other GPU-bound code already noted elsewhere in this codebase as untestable here), not something this stage introduced. Coverage was placed instead on the pure-math layer plus the controlled-callback widget tests above.
