## Spike A — FFI-embedded solver feasibility

**Question this answers:** can the actual SolveSpace C++ solver core (the library `py-slvs` wraps) run in-process on the client via Dart FFI, instead of over HTTP — reusing the proven solver rather than reimplementing it?

### Steps

1. **Source the library.** Check both `realthunder/slvs_py` (the fork this project's backend already depends on, SolveSpace v2.3-based, Linux-memory-optimized) and upstream SolveSpace's own `libslvs` — the fork's Linux-specific optimizations may not matter for a mobile target, and upstream's build may have broader existing cross-platform support already. Confirm which one is the better starting point before building anything.
2. **Build for `arm64-v8a` via the Android NDK**, targeting the actual test device architecture (Adreno 740 / Snapdragon 8 Gen 2, per the existing on-device test setup) — independent of Python and SWIG entirely. Output: a `libslvs.so`/`.a` with no Python in the loop. **Revised scope (see below): also scope the equivalent iOS `arm64` build (Xcode toolchain, static `.a`/XCFramework) alongside Android** — not full parity work, just confirming the same CMake source cross-compiles there too, since Android and iPhone UX for this app are expected to converge closely enough that iOS shouldn't be de-risked separately later as an afterthought.
3. **Establish an FFI-callable surface** for the primitive set this project actually uses today (from `constraints.py`'s own table): `addPointsDistance`, `addPointsProjectDistance`, `addPointsVertical`/`Horizontal`, `addAngle`, `addPointsCoincident`, `addParallel`/`addPerpendicular`, `addEqualLength`, `addPointLineDistance`, `addPointOnLine`, `addMidPoint`, `addCurvesTangent`, plus group/point creation, solve, param readback, `Dof`, and the failed-constraint list. If the library's C++ API isn't already flat/C-ABI-compatible, scope how large a thin C shim would need to be — don't assume one is trivial, but don't assume it's large either until it's actually looked at.
4. **Correctness parity check.** Pick one simple case (two points + one distance constraint) and one hard case (a Slot — chosen deliberately because it's the most structurally fragile shape in the current system, per `docs/sketcher-architecture-ux-scoping.md` §4.5). Solve both through the new FFI path and through the existing backend's `/solve` endpoint with identical inputs; confirm matching `dof`/`converged`/point-position results.
5. **Dart integration check.** Confirm `dart:ffi` loads and calls the compiled library from a real Flutter Android build (not a bare script) — including the actual packaging step (bundling the `.so` into the APK).
6. **Licensing.** The solver is GPL-3. Get a real, specific answer on what obligations attach to compiling it into a distributed client app (even a free/friends-and-colleagues distribution) — static vs. dynamic linking may matter here. This needs an actual informed read, not an assumption in either direction; flag for a proper legal check if a confident answer isn't reachable from the license text alone. **Revised scope: check this separately for each distribution channel** — Android's friends-and-colleagues distribution model (sideloaded APK, no store DRM) and iOS's (which realistically means the App Store, or at minimum Xcode/TestFlight-signed builds) may not land on the same answer; see the iOS-specific finding below.

### Revised scope note (iOS)
The original scoping treated iOS as out of scope, reasoning that Android is the real target device and the cheapest toolchain to stand up first. That's still true for *sequencing* — Android stays first, since the physical Adreno 740 test device is Android. But iOS should no longer be dropped from consideration entirely: this app's Android UX is expected to be close enough to an equivalent iPhone UX that de-risking iOS later, as an afterthought, would be the wrong call — better to at least confirm nothing about the FFI/build approach is Android-specific while this spike is already in motion. Concretely: iOS build feasibility (step 2) and iOS-specific licensing (step 6) are now in scope for this spike as research/scoping, not full parity; iOS on-device testing (there's no physical iOS test device yet) and iOS UI work stay out of scope, same as before.

### Explicitly out of scope
Windows builds (no current target device or user need); full iOS on-device testing (no physical device yet — build feasibility and licensing are now scoped, actual on-device verification is not); porting the full constraint vocabulary (the primitive list above is the target, not "everything"); any UI or interaction work; touching the existing backend.

### Go / no-go
- **Go** (proceed to restructure the sketcher around an in-process solver) if: the library builds clean for `arm64-v8a` Android, the FFI surface is clean or the shim is small, the Slot case matches the backend's result, Dart/Flutter integration works end to end, and licensing is clear and acceptable **for the Android distribution path**. iOS build feasibility and licensing should be reported alongside this even if not yet a hard gate, so a future iOS de-risking pass starts from evidence instead of zero.
- **No-go / park** if any of build, FFI surface, or licensing turns out to be a hard blocker on Android. Falls back to the original incremental plan (`docs/sketcher-restructure-plan.md` Phases 0–2: round-trip reduction, decoupled drag rendering, Slot as a real backend entity). If Android clears but iOS's licensing path looks meaningfully worse (see finding below), that's a separate, explicit call to make later — not a reason to block the Android "Go."
- **Licensing question decided (2026-07-16):** GPL-3 is accepted, not a remaining go/no-go gate — see the "Decision" note above and the "Licensing implications" finding below. The technical build/FFI/parity criteria above remain live gates; licensing is not.
- **VERDICT (2026-07-16): GO, on Android, confirmed on real hardware.** Every technical gate above is now met for real, not by plausibility argument: clean `arm64-v8a` build, a small precisely-bounded shim (27 functions, all built and exported correctly), Slot parity confirmed at the raw solver level (exact match), and `dart:ffi` loading + solving confirmed inside a real release APK installed and run on the actual Adreno 740 test device. See "Track 1 verdict — real on-device execution" below for the full account, including one real caveat the Go verdict carries forward: a chunk of interpretive business logic (redundancy handling, provisional-DOF flooring) lives in Python today and isn't part of the raw solver — it will need porting to Dart alongside the constraint-construction calls, not assumed to come for free.

## Research-pass findings (2026-07-16)

Everything below was done from a cloud sandbox with **no Flutter SDK, no Android NDK, no `adb`, and no physical or emulated device** — so it covers only the research-shaped sub-steps of each spike (library sourcing, license text, header/API inspection, package-source reading). The build- and on-device-shaped steps (A2, A4, A5, B1) are untouched and need an environment with those tools. GitHub access in this session is also scoped to `didsa-uk/didsa-cad` only, so external repos below were read via raw file fetches and pub.dev's own API rather than GitHub's API/UI.

### Spike A

**Step 1 — source comparison.** The backend's actual pinned dependency is `py-slvs==1.0.6` (`backend/environment.yml`), which is [realthunder/slvs_py](https://github.com/realthunder/slvs_py) — a SWIG-generated Python binding over a fork of SolveSpace v2.3 with Linux-specific memory optimizations, GPL-3, also used as FreeCAD Assembly3's solver. Upstream `solvespace/solvespace` carries its own `libslvs/` subdirectory with an independent `CMakeLists.txt`, and the root build already supports disabling the GUI/CLI (`ENABLE_GUI`, `ENABLE_CLI` CMake options) and cross-compiling (a documented MinGW toolchain-file path from Linux) — no Android/NDK toolchain file exists today, but the build is plain CMake/C++ with no Python or SWIG step, so adding an NDK toolchain file is the standard CMake cross-compile pattern, not a new build system. Not yet confirmed: whether realthunder's fork carries the same clean `libslvs/` split or any Linux-specific code that would need adjusting for Android — that needs the fork's own tree checked out, not just its README.

**Step 3 — FFI surface.** Fetched upstream's actual `include/slvs.h` directly (raw.githubusercontent.com). It is **already a flat, `extern "C"`-wrapped C-ABI header** — `Slvs_hParam`/`Slvs_hEntity`/`Slvs_hConstraint` handles are typed `uint32_t`, and it exposes convenience constraint functions matching most of the project's `SolverBuilder` protocol directly by name: `Slvs_Coincident`, `Slvs_Distance`, `Slvs_Equal` (equal length), `Slvs_Parallel`, `Slvs_Perpendicular`, `Slvs_Midpoint`, `Slvs_Angle`, `Slvs_Tangent`, `Slvs_EqualPointToLine` (the `SLVS_C_EQ_LEN_PT_LINE_D` primitive `equal_length_point_line_distance` needs), `Slvs_Horizontal`/`Slvs_Vertical`, plus entity constructors (`Slvs_AddPoint2D`, `Slvs_AddLine2D`, `Slvs_AddCubic`, …) and `Slvs_Solve`/`Slvs_SolveSketch`/`Slvs_GetParamValue`. Two primitives this project needs (`point_on_line` → `SLVS_C_PT_ON_LINE`, `curves_tangent` → `SLVS_C_CURVE_CURVE_TANGENT`) have no dedicated convenience wrapper in this header but are reachable through the generic `Slvs_AddConstraint(grouph, type, …)` entry point using those constraint-type enums directly. Net: **a dedicated C shim looks unnecessary, or at most a few lines** — `dart:ffi` can plausibly bind straight to this header's functions. Caveat carried forward: this is *upstream's* header; it hasn't been confirmed identical to the header realthunder's fork actually ships (the fork predates some upstream additions), so this is a strong signal, not proof, until the fork's own `slvs.h` is diffed against it.

**Step 6 — licensing.** Quoting the GNU Project's own GPL FAQ directly: linking a GPL-covered work "statically or dynamically with other modules is making a combined work based on the [GPL-covered work], and the terms and conditions of the GNU General Public License cover the whole combination" — GPL-3 draws **no distinction between static and dynamic linking**; both trigger the same copyleft obligation on the combined work. (This is unlike LGPL, where static vs. dynamic does matter.) Practical read for this spike: shipping a client app that links this solver, by either method, would put the combined app under GPL-3's terms — meaning source availability obligations to whoever the app is distributed to, "friends and colleagues" included. This is a real, non-trivial constraint on the "Go" path, not a formality — flagged here as confirmed from the license text itself, but still worth the "proper legal check" the spike doc already calls for before treating it as fully settled for this project's specific distribution plans.

**Not executable in this environment:** A2 (NDK build for `arm64-v8a`), A4 (on-device Slot parity vs. the backend `/solve` endpoint), A5 (Dart FFI + real Flutter Android APK packaging). These need a machine with the Android NDK, a Flutter SDK, and either the physical Adreno 740 test device or an equivalent emulator.

**iOS scope addition — build feasibility.** Upstream `solvespace/solvespace`'s repo ships a MinGW cross-compile toolchain file but, confirmed by direct fetch, no iOS-specific one (`cmake/ios.cmake`/`Toolchain-iOS.cmake` and similar names all 404 against the repo). That's not a red flag specific to this project, though: cross-compiling plain CMake C/C++ libraries for iOS is a well-worn, generic pattern with mature community tooling (e.g. `leetal/ios-cmake`, actively maintained, used widely for exactly this "build an existing CMake C library as an iOS static lib/XCFramework" case) — the same category of "add a toolchain file" work the MinGW case already demonstrates upstream is comfortable with, not a new build system. Since `libslvs` is self-contained C/C++ with no platform-specific GUI dependencies once `ENABLE_GUI`/`ENABLE_CLI` are off, there's no structural reason visible from the source that it wouldn't cross-compile for iOS `arm64` the same way it would for Android `arm64-v8a` — but this is a plausibility read from the CMake structure, not a confirmed build; nobody has actually run it.

**iOS scope addition — licensing (the more material finding).** The general GPL-3 combined-work obligation from Step 6 above applies identically on iOS. But iOS distribution adds a documented, iOS-specific complication that Android's sideload/friends-and-colleagues model doesn't have: **Apple's App Store wraps all downloads in its own DRM (FairPlay), which directly conflicts with the GPL's guarantee that recipients can freely redistribute the software.** This isn't a theoretical concern — it's exactly what happened to VLC (also GPL-licensed): the GPL copyright holder had it pulled from the App Store in 2010–2011 specifically because Apple's store terms and GPLv2's redistribution terms were incompatible, and Apple's response was to remove the app rather than resolve the conflict. (VLC later returned to the App Store, but under arrangements outside plain "GPL app in the store" — not something to assume solves this for a different codebase without checking.) Practical read: for Android, sideloaded/friends-and-colleagues distribution sidesteps any store-DRM conflict entirely, so the Step 6 "combined work is GPL-3" finding is the whole story. For iOS, *even if* the build and FFI work cleanly, App Store distribution specifically has a real, precedented licensing obstacle beyond the general GPL-combined-work point — direct developer-to-device distribution (Xcode debug builds, TestFlight, or an enterprise certificate) would avoid the App Store's DRM wrapping the same way Android sideloading does, so the conflict is about *the App Store specifically*, not iOS as a platform. This should be treated as a genuinely open legal question for the professional legal check the doc already calls for, not resolved here.

**PlaneGCS alternative — evaluated and rejected on technical merits, independent of licensing.** Before deciding to accept GPL-3, `PlaneGCS` (FreeCAD's own 2D geometric constraint solver, LGPL-2.0-or-later, confirmed via its actual header `FreeCAD/FreeCAD`'s `src/Mod/Sketcher/App/planegcs/GCS.h`) was checked as a licensing escape hatch. It would genuinely have solved the GPL problem — LGPL's dynamic-linking exception doesn't carry GPL-3's "no distinction between static and dynamic" rule, and an open-sourced DIDSA-CAD client would satisfy LGPL's "user can relink a modified copy" requirement even on locked-down mobile OSes (the standard real-world pattern, e.g. FFmpeg's LGPL build variant used specifically to enable App-Store-safe apps). But disregarding licensing, it loses the technical comparison: its `GCS::System` API is a C++ class with overloaded methods taking C++ reference types (`Point&`, `Line&`, …), not a flat C-ABI, so it needs a real FFI shim where SolveSpace needs almost none; and while its diagnostic design is more rigorous on paper (separate conflicting/redundant/partially-redundant constraint tracking via QR decomposition of the Jacobian, vs. SolveSpace's blunt "return every constraint in an inconsistent system"), independent sources describe SolveSpace's solver specifically as unusually stable and predictable, while FreeCAD's Sketcher (which is what actually exercises `PlaneGCS` in production) has a long-running, currently-open history of false redundant/conflicting-constraint detection bugs — the same failure class (Slot, Tangent/EqualRadius redundancy) this project has already spent real effort taming in `constraints.py`. Switching solvers would mean rediscovering equivalent workarounds against a different engine with its own open reliability issues, plus reintroducing a client/server parity risk unless the backend migrated too. Verdict: not worth it on technical grounds alone — the decision above to keep SolveSpace and accept GPL-3 is a deliberate trade, not a default.

**Licensing implications of the GPL-3 decision.** GPL-3 involves **no royalties and no upfront licensing fee** — that's not how copyleft works; there's no payment owed to SolveSpace/py-slvs's authors for using, modifying, or distributing their code, and GPL-3 explicitly does not cap what DIDSA-CAD can charge its own users for distributing copies (the GNU project's own philosophy page: "you can charge nothing, a penny, a dollar, or a billion dollars"). The real obligations are behavioral, confirmed against GPL-3's actual text:
- **Source disclosure**: anyone who receives a DIDSA-CAD binary must also receive (or get a written offer for) the complete corresponding source of the whole combined app. Already addressed by the decision to open-source the repo.
- **The whole app inherits GPL-3 for recipients**, including their own right to redistribute — modified or not, for free. This is the actual commercial risk (someone could give away what a customer paid for), not a fee owed to anyone.
- **No additional restrictions** on top of GPL-3's own terms — this is the clause the Apple App Store's DRM terms conflict with (finding above).
- **Anti-Tivoization (GPLv3-specific)**: conveying the software as part of a "User Product" (consumer devices for personal use — a phone qualifies) requires providing "Installation Information" sufficient for a user to install and run a *modified* version on that same device. Android already permits sideloading/re-signing, so this isn't a new constraint in practice there; on iOS it's moot, since the App Store path is already blocked for the separate DRM reason above.
- **Copyleft propagation**: any modifications DIDSA-CAD itself makes to the SolveSpace/py-slvs code must also be GPL-3 if distributed (no obligation to upstream them to the SolveSpace project itself — that's a courtesy, not a license term).
- **Permanent per copy conveyed** — can't be revoked retroactively for recipients who already have a copy.

**Distribution channel implications.** The Apple-App-Store-specific DRM conflict above is genuinely specific to Apple's own store terms, not to iOS as a platform or to GPL generally — **Google Play has no equivalent conflict** (Android permits sideloading and doesn't wrap apps in comparable anti-redistribution DRM; GPL apps, including VLC for Android, have shipped there without controversy throughout). Since the physical test device and near-term focus are already Android, accepting GPL-3 gives a clean path to **Google Play today**. Reaching iPhone customers without the App Store is real but meaningfully harder than the Android equivalent, not a like-for-like substitute: Ad-Hoc distribution caps at 100 registered device UDIDs/year (not a real customer channel); Apple's Enterprise Program is licensed only for internal/employee distribution, so using it to sell to the public would violate Apple's own terms independent of GPL; TestFlight is still Apple-run infrastructure under the same Developer Program terms and isn't confirmed to escape the conflict; AltStore-style resigning tools work but require periodic re-signing and are a clunky, non-mainstream experience; the EU's Digital Markets Act now forces Apple to permit alternative marketplaces/direct install, but only in the EU, and Apple's notarization terms for that channel haven't been tested against GPL compatibility. Net: treat iOS as **not a viable near-term distribution target** under this decision, rather than assuming "direct distribution instead of the App Store" is an easy equivalent door — Google Play is the actual clean path this decision unlocks.

**What other developers actually do instead of building their own solver.** Checked what the wider ecosystem does when facing this same GPL-3 dilemma, rather than assuming "write a replacement" is a live option: it mostly isn't. A direct search for free/permissive solvers comparable to SolveSpace turned up none — this project's own restructure plan already rejected reimplementing one in Dart for the same reason (§2: "real risk of reintroducing bugs already fixed server-side"), and the ecosystem search independently confirms nobody else has successfully done it either.
- **Same-scale precedent (what indie developers embedding SolveSpace's actual solver do):** [`hlorus/CAD_Sketcher`](https://github.com/hlorus/CAD_Sketcher), a Blender add-on doing the same kind of embedding Spike A wants, is itself licensed GPL-3 — confirmed directly from its repo. This is the standard route at this project's scale, and it's the one already chosen here.
- **What well-funded commercial CAD vendors do instead — license a proprietary solver, for real money.** Confirmed Shapr3D (a real, successful paid App Store CAD app) uses Siemens' **D-Cubed 2D DCM** for sketch constraint solving (paired with Parasolid for the 3D kernel) — the same D-Cubed stack SolidWorks, Onshape, Inventor, NX, and others license. This sidesteps GPL by not using open-source code at all, in exchange for real OEM/enterprise-tier licensing fees (Siemens doesn't publish pricing, but it's negotiated commercial licensing, not a one-time indie fee). **Your call: not realistic at this project's current scale, kept on record only as a break-glass option if DIDSA-CAD's circumstances change dramatically (e.g. real commercial traction or outside investment) — not something to plan around now.**
- **A genuinely cheap, unexplored option: contact SolveSpace's own maintainer(s) directly about a non-GPL license for this specific use.** SolveSpace's own licensing guidance apparently already invites exactly this inquiry for applications that need different terms. Costs nothing but an email, and could remove the GPL question entirely if granted. **Your call: worth trying, but sequenced *after* testing real demand on Android under the GPL-3/Google Play path already decided above — not a prerequisite to starting, and not worth spending the ask on a project that hasn't yet shown it has real users.**

### Spike B

**B2 — fully answered by this pass; this is the headline finding.**
- Re-checked `flutter_scene`'s pub.dev listing via its own API (`https://pub.dev/api/packages/flutter_scene`): the current published latest is **0.18.1**, exactly what `client/pubspec.yaml` already pins. Nothing has changed since it was last checked.
- Pulled the actual published package source (via pub.dev's archive API, since this session's GitHub access doesn't extend to `bdero/flutter_scene`) and read `lib/src/camera.dart` and `lib/src/render/scene_pass.dart` directly. Finding: **`Camera` is already an abstract base class that separates *view* (`getViewMatrix()`, `position`/`forward`/`up`) from *lens* (a `projection` getter returning a separate abstract `CameraProjection`)**. `CameraProjection`'s only built-in concrete implementation is `PerspectiveProjection`, but its own doc comment says outright: *"applications can implement `CameraProjection` for orthographic or other projections."* Grepped the whole package source for hardcoded `PerspectiveCamera` type-checks anywhere in the render pipeline (`scene_pass.dart`, `ssao_pass.dart`, the encoder/culling code) — found none; everything downstream consumes the generic `Camera`/`CameraProjection` interface (`getViewTransform`, `Frustum.matrix`, etc.), which is projection-agnostic.
- **This directly answers `docs/didsa-longterm-vision-and-model.md` §13's stated gate** ("checking whether `flutter_scene`'s `MakeOrthographic` factory is already surfaced at the Dart level"): there is no factory literally named `MakeOrthographic`, but the more general mechanism that would make one unnecessary — a pluggable `CameraProjection` interface, explicitly documented as extension-ready for orthographic — already exists in the exact version this project has pinned.

**Step 4/4b — on-device parity check, real result.** Built a throwaway Flutter app (`slvs_ffi_harness`), bundled the shim `.so` under `android/app/src/main/jniLibs/arm64-v8a/`, and ran the two test cases via `dart:ffi` directly on the device. - **Simple case (two points + one distance constraint): exact match.** `converged=true, dof=3, result_code=0`, point positions bit-identical between the on-device FFI path and the backend.
- **Slot case: exact match at the raw solver level, with a real and important nuance surfaced.** Both paths report `result_code=5` (`REDUNDANT_OK`) and identical point positions to 4 decimal places. The on-device harness initially printed `converged=false, dof=0` against the backend's `converged=true, dof=1` — traced this precisely rather than treating it as a solver bug, by reading `solver.py`'s own code: both figures are **the result of deliberate Python-side business logic that doesn't exist in the raw solver at all**:
  1. A narrow override forces `converged=True` when the result code is 4/5 *and* every constraint in the sketch is drawn from a specific safe list (`DistanceConstraint`, `VerticalConstraint`, `HorizontalConstraint`, `AngleConstraint`, `CoincidentConstraint`, `ParallelConstraint`, `PerpendicularConstraint`, `EqualLengthConstraint`, and others including Tangent/EqualRadius) — deliberately excluding `AtMidpointConstraint`, because a documented test (`test_stage15_constraints.py`) proves the same result code can also mean a genuinely under-constrained shape in that specific case.
  2. A "provisional DOF floor": when converged (after the above) and a still-unconfirmed (`provisional`) `DistanceConstraint` is present — exactly this Slot's unconfirmed radius — `dof` is bumped to `max(raw_dof, 1)` to compensate for a documented py-slvs undercounting quirk in this exact scenario.
  
  Once the on-device harness's raw `Dof` reading (0) is compared against what these code comments say the backend's own *raw* `system.Dof` is for this exact construction (also 0, confirmed from `solver.py`'s own inline documentation of this precise case) — **the raw solver-level result is a genuine, exact match.** The reported `SolveResult.converged`/`.dof` values only differ because of interpretive logic layered on top in Python.

  **This is the load-bearing finding for scoping any real migration**: an in-process Dart solver would need to port this redundancy-classification and DOF-flooring logic too, not just the constraint-construction calls — the raw solver alone reproduces correct geometry but not today's UX (e.g. a fresh Slot correctly *not* showing as fully constrained until its radius is confirmed).

**Step 5 — real Flutter/APK integration: done, as part of Step 4's own harness, not separately.** The harness *is* a real Flutter Android app, built via `flutter run --release` (which produces and installs a genuine `app-release.apk`, not a debug/dev artifact), with the shim `.so` bundled the same way a production app would (`jniLibs/arm64-v8a/`). `DynamicLibrary.open('libslvs_ffi.so')` loaded and solved correctly on the real device once the static-linking fix landed.
